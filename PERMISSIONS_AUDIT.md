# מיפוי הרשאות מלא במערכת (כולל מובייל)

## מתודולוגיה קצרה
- המיפוי בוצע לפי קוד בפועל במסכים: `index.html`, `app.js`, `dashboard.js`, `scheduling.html`, `schedulingsystem.html`, `scheduling-app.js`.
- הבדיקה כוללת גם הרשאות UI (מה מוצג) וגם חסימות לוגיות (מה באמת נחסם/מותר בקוד).
- סטטוסים:
  - **מותר** = פעולה זמינה ומאושרת לוגית.
  - **אסור** = פעולה חסומה לוגית.
  - **חסום חלקית** = מוסתר/מנוטרל ב-UI אך חסימה לוגית חסרה או חלשה.

## 1) סוגי משתמשים שזוהו בפועל

| סוג משתמש | איך מזוהה | מקור בקוד/דאטה | הערות |
|---|---|---|---|
| משתמש לא מחובר | אין `dash_hash`/`dash_role` תקפים, או טרם התחברות | `app.js` (`showLogin`, `resumeSession`) | רואה רק מסך כניסה |
| מדריך (`instructor`) | התחברות דרך קובץ `data/instructors/<hash>.json` | `app.js` (`login`) | ננעל למצב חודש בלבד |
| מנהל (`admin`) | התחברות דרך קובץ `data/admins/<hash>.json` | `app.js` (`login`) | גישה למסכי ניהול שונים |
| תפקיד כפול (`both`) | קובץ admin עם `role === 'both'` | `app.js` (`startApp`) | יכול לעבור בין תצוגת מנהל/מדריך |
| מנהל עם הרשאת תאריכי סיום מורחבת | לפי `EmployeeID` = `7000` או `8000` | `dashboard.js` (`updateEndDatesButtonVisibility`) | הרשאה נקודתית למסך תאריכי סיום |
| משתמש עם הרשאת שיבוץ ZOOM | לפי `EmployeeID` = `6000` או `8000` | `dashboard.js` (`canAssignZoom`) | הרשאה לפעולות שיבוץ/שמירה ב-ZOOM |
| משתמש עם הרשאת "מצא שיבוץ" במערכת Scheduling | לפי `EmployeeID` = `6000` או `8000` | `scheduling-app.js` (`SCHEDULING_AUTHORIZED_IDS`) | גם אם admin – לא בהכרח מורשה להריץ מנוע שיבוץ |

> לא נמצאו בקוד תפקידים "תלמיד/משתתף" או "משתמש משחק" נפרדים. לא נמצאו ישויות "חדר", "PLAY", "RESET" במערכת זו.

## 4) בדיקות לוגיות (UI Permission vs Actual Permission)

### ממצאים עיקריים
1. **תאריכי סיום – חסימה ויזואלית בלבד (חסר guard לוגי מלא).**
   - הכפתור מוסתר לפי EmployeeID, אבל המעבר למצב `enddates` לא מבצע בדיקת הרשאה נוספת.
   - משתמש admin לא מורשה יכול תאורטית להפעיל `window.mode='enddates'; render();` דרך קונסול/סקריפט.

2. **Scheduling תלוי role ב-storage בצד לקוח.**
   - guard בעמוד `scheduling.html` נשען על `sessionStorage/localStorage`.
   - ניתן לזייף role בצד לקוח (אם אין אימות שרת מאחורי המשאבים/פעולות).

3. **הקשחת הרשאות חלקית טובה ב-ZOOM וב-Scheduling run.**
   - יש בדיקה כפולה גם ב-UI וגם בלחיצה (`canAssignZoom`, `SCHEDULING_AUTHORIZED_IDS`).
   - עדיין חסר אכיפה צד שרת כדי למנוע עקיפה מלאה.

## 5) פערים ביחס לדרישת "חדר / PLAY / RESET / קוד חדר"

לא נמצאו במערכת רכיבים או פעולות בשם:
- יצירת חדר,
- PLAY,
- RESET,
- מזהה חדר,
- קוד חדר,
- סטטוס מערכת משחק.

כלומר, הדרישה הספציפית הזו אינה ישימה ישירות לריפו הנוכחי (לפחות בקבצים שנבדקו). אם קיימת מערכת משחק חיצונית/ריפו נוסף — יש למפות אותה בנפרד.

## 6) המלצת מבנה הרשאות מסודר (RBAC)

1. **מדיניות מרכזית אחת** (למשל `permissions.js`):
   - `canViewEndDates(user)`
   - `canAccessZoom(user)`
   - `canAssignZoom(user)`
   - `canAccessScheduling(user)`
   - `canRunScheduling(user)`

2. **אכיפה כפולה חובה לכל פעולה רגישה**:
   - UI: הסתרה/ניטרול כפתור.
   - Logic: guard בפונקציית הפעולה עצמה.

3. **אכיפת שרת לפעולות write/risky**:
   - שמירת ZOOM assignments.
   - הרצת scheduling.
   - כל פעולה המשנה נתונים.

4. **אחידות מובייל/דסקטופ**:
   - הרשאה לא תשתנה בגלל viewport.
   - רק אופן ההצגה משתנה.

5. **Audit Trail**:
   - לוג משתמש/תפקיד/פעולה/תוצאה לפעולות רגישות.

## 7) רשימת בעיות קריטיות מסומנות

### קריטית A — הרשאה חסרה לוגית במסך תאריכי סיום
- **מה הבעיה:** אין חסימה לוגית מלאה, רק הסתרת כפתור לפי מזהה עובד.
- **איפה:** `dashboard.js` (`btnEndDates.onclick`, `renderEndDates`).
- **עבור מי:** admin שאינו 7000/8000 (או מי שמצליח לשנות state).
- **גם במובייל?:** כן (state logic זהה).
- **תיקון נכון:** guard מפורש לפני `window.mode='enddates'` ובתוך `renderEndDates`.

### קריטית B — הרשאת Scheduling תלויה storage
- **מה הבעיה:** guard גישה מבוסס `dash_role` ב-local/session storage.
- **איפה:** `scheduling.html`.
- **עבור מי:** כל משתמש שיכול לתמרן storage בדפדפן.
- **גם במובייל?:** כן.
- **תיקון נכון:** אימות שרת חתום/מאומת ולא רק state מקומי.

### קריטית C — אכיפה צד לקוח בלבד בפעולות רגישות
- **מה הבעיה:** הרשאות כמו ZOOM assign / Scheduling run מסתמכות על JS לקוח.
- **איפה:** `dashboard.js`, `scheduling-app.js`.
- **עבור מי:** משתמשים מתקדמים/תוקפים בצד לקוח.
- **גם במובייל?:** כן.
- **תיקון נכון:** ולידציה בשרת לכל POST/פעולה רגישה.

## 8) סיכום ניהולי קצר
- המערכת מיישמת חלוקת תפקידים בסיסית טובה (instructor/admin/both), עם מנגנוני UI ברורים.
- קיימים חריגים שבהם ההסתרה ב-UI אינה מספיקה (בעיקר End Dates).
- במובייל יש הבדלי תצוגה משמעותיים (למשל הסתרת פילטרים), אך רובם UX ולא פרצת הרשאה.
- כדי להגיע לרמת אבטחה תפעולית גבוהה: להעביר החלטות הרשאה קריטיות לצד שרת + שכבת RBAC מרכזית משותפת לכל המסכים.
