# מסמך היכרות למערכת DASH (עבור ChatGPT)

## מטרת המערכת במשפט אחד
מערכת DASH היא דשבורד ניהול והצגת פעילויות/קורסים עם התחברות לפי תפקיד, ובנוסף מודול שיבוצים חכם שמייצר המלצות שיבוץ מתוך `scheduling.json`.

## מבנה הפרויקט / קבצים מרכזיים
- `index.html` – מעטפת ה-UI הראשית (מסך כניסה + דשבורד) עם רוב ה-CSS המוטמע.
- `app.js` – לוגיקת כניסה, אימות משתמש, טעינת נתונים לפי הרשאה והפעלת הדשבורד.
- `dashboard.js` – כל לוגיקת התצוגות של הדשבורד (חודשי/שבועי/סיכום/מדריכים/תאריכי סיום), פילטרים, חישובים עסקיים.
- `scheduling.html` – שער גישה לעמוד שיבוץ (כולל בדיקת הרשאה לפי מזהי עובד).
- `schedulingsystem.html` – ממשק מודול השיבוץ החכם.
- `scheduling-app.js` – אתחול מודול השיבוץ, טעינת `scheduling.json`, הפעלת מנוע ההמלצות והצגת תוצאות.
- `core/*.js` – מנועי עזר לשיבוץ (המלצות/חוקים/ניקוד/טעינה/נסיעות/תאריכים/הסבריות).
- `generator.html` – מחולל נתונים: קולט קבצי Excel ומייצר ZIP עם כל קבצי ה-JSON הנדרשים.
- `data/` – תיקיית נתונים בפורמט JSON (admins / instructors / Scheduling).
- `feature-flags.js` – פיצ'ר-פלאגים (בעיקר הפעלת/כיבוי השיבוץ).
- `server.js` – שרת Node קטן להגשה סטטית מקומית (ללא Cache).

## אילו קבצי HTML קיימים ומה כל אחד עושה
- `index.html`: עמוד ראשי של המערכת (RTL), כולל כניסה, ניווט וכל אזור הדשבורד.
- `generator.html`: כלי אדמין לייצור נתונים (Excel → JSON בתוך ZIP).
- `scheduling.html`: שכבת Guard; מוודא שהשיבוץ מאופשר + מזהה משתמש מורשה (6000/8000) לפני מעבר למערכת שיבוצים.
- `schedulingsystem.html`: ממשק העבודה של מודול השיבוץ (בחירת רשות יעד, משך קורס, Top N, תוצאות).

## אילו קבצי JS/CSS מרכזיים קיימים ומה התפקיד שלהם
- `app.js`:
  - Hash התחברות (`SHA-256`) לפי נוסחת `EmployeeID + EntryCode + SALT`.
  - חיפוש קובץ משתמש קודם ב-`data/instructors/` ואז ב-`data/admins/`.
  - ניהול session/local storage + Resume Session.
- `dashboard.js`:
  - נרמול נתונים מאקסל (תאריכים/שעות), מצבי תצוגה, KPI, מודאלים, פילטרים.
  - חישובי “חסר מדריך”, עומסי מדריכים, תאריכי סיום ועוד.
  - הרשאות תצוגה לפי תפקיד (instructor/admin) ולפי מזהי עובד מיוחדים.
- `scheduling-app.js`:
  - טעינת `data/Scheduling/scheduling.json`.
  - קריאה ל-`buildGlobalRecommendations` והזרקת תוצאות UI.
- `core/RecommendationEngine.js`:
  - ליבת בחירת מועמדים לשיבוץ (טווח תאריכים, חלונות זמן, מרחקים, ניקוד).
- `core/RenderEngine.js`:
  - רינדור רשימת המלצות השיבוץ.
- CSS:
  - רוב הסטייל מוטמע ישירות ב-`index.html`, `generator.html`, `schedulingsystem.html`.
  - `style.css` קיים בפרויקט אך בפועל לא מיובא ישירות מה-HTML המרכזיים.

## הדאטה והזרימה
### מאיפה מגיעים הנתונים
הזרימה בפועל היא:
1. מעלים ל-`generator.html` שלושה קבצי Excel (`main.xlsx`, `codes.xlsx`, `distance.xlsx`).
2. המחולל יוצר ZIP שמכיל:
   - `data/instructors/<hash>.json`
   - `data/admins/<hash>.json`
   - `data/Scheduling/scheduling.json`
3. את ה-ZIP פורסים לתוך הפרויקט (או על שרת הסטטי), והמערכת טוענת JSON בזמן ריצה.

### שמות/תיקיות הנתונים
- `data/instructors/` – קובץ JSON לכל מדריך (לפי hash התחברות).
- `data/admins/` – קובץ JSON לכל אדמין (לפי hash התחברות).
- `data/Scheduling/scheduling.json` – מקור הנתונים של מודול השיבוץ וגם metadata נוסף לדשבורד.

### מבנה כללי של אובייקט “קורס/אירוע” והשדות הקריטיים
אובייקט קורס/אירוע (במיוחד ב-`scheduling.json`) כולל בדרך כלל:
- מזהים: `EmployeeID`, `Employee`
- סיווג: `EventType` (למשל `COURSE`, `HOLIDAY`)
- הקשר: `Program`, `Authority`, `School`
- ניהול: `CourseManager`, `InstructorManager`
- זמנים: `StartTime`, `EndTime`, `Dates[]`, `End`, `MonthStart`, `MonthEnd`
- שדות עזר מתהליך האקסל: `Date1`…`Date16` (ובנוסף `Dates[]` מאוחד)

ובאובייקט מדריך לשיבוץ:
- `EmployeeID`, `Employee`, `Role`
- `EmploymentType`
- `HomeAuthority`
- `InstructorManager`

שדות קריטיים במיוחד ללוגיקת שיבוץ:
- `Dates`, `StartTime`, `EndTime`
- `EmployeeID`, `EventType`
- `Authority`, `HomeAuthority`
- `EmploymentType` (לצורכי תצוגה/סינון עסקי בדשבורד)

## הרשאות ותפקידים
- **instructor**:
  - רואה דשבורד מצומצם (בעיקר תצוגת לוח אישית).
  - לא רואה כפתורי Summary/Instructors/Month/Week ומסננים מתקדמים.
- **admin**:
  - רואה את כלל תצוגות הניהול והפילטרים.
- **מזהים מיוחדים**:
  - `6000` או `8000`: יכולים לראות/להיכנס לכפתור שיבוץ (`שיבוץ`) כאשר הפיצ'ר פעיל.
  - `7000` או `8000`: יכולים לראות כפתור `תאריכי סיום`.
- **Feature Flag**:
  - אם `ENABLE_SCHEDULING=false`, עמודי השיבוץ מוסתרים/נחסמים גם למורשים.

## לוגיקות חשובות שכדאי להכיר
### חישוב “חסר מדריך”
“חסר מדריך” מזוהה כאשר:
- `EventType === COURSE`
- שדה `Employee` ריק/לבן
- והקורס פעיל בחודש הנבחר **או** מוגדר להיפתח בעתיד (לפי תאריך התחלה).

### טווח הצעות שיבוץ (Today + X ימים)
בליבת ההמלצות (`RecommendationEngine`) הטווח הוא:
- מהיום +7 ימים ועד מהיום +30 ימים.
- רק ימים עם `getDay()` בין `0..4` (ראשון–חמישי).

> הערה: בממשק מופיע גם שדה “כמות ימים להציג”, אבל בפועל הליבה משתמשת בטווח הקבוע בקוד ההמלצות.

### כללים עסקיים עיקריים במנוע השיבוץ
- שעות עבודה לחיפוש חלון: `08:00–15:00`.
- אם למדריך כבר יש 3 אירועים באותו יום – נפסל.
- חייב להיות חלון פנוי באורך משך הקורס.
- ולידציית רצף נסיעות/מרחקים בין אירועים באותו יום:
  - אותו יעד/אזור: מגבלות מרחק שונות מאזורים שונים.
  - בין רשויות: פסילה במרחק גבוה (לפי תנאים בקוד).
- בדיקת קונפליקט עתידי לפי אותו יום בשבוע בטווח 60 יום קדימה.
- ניקוד משקלל מרחק מהבית, עומס יומי/שבועי/חודשי, ועומס עתידי.

## כללי UI/עיצוב שכבר קיימים
- RTL מלא (`lang="he" dir="rtl"`) בכל ה-HTML המרכזיים.
- עיצוב “כרטיסים” מודרני עם צבעים בהירים, כפתורים בולטים ושפה חזותית אחידה.
- מובייל: דגש על Tap targets, אנימציות לחיצה, קומפקטיות טבלאות.
- בדשבורד יש sticky header/nav במובייל ושימוש ב-accordion/כרטיסים בתצוגות שונות.
- תיבות חיפוש וטבלאות קיימות כבר (לדוגמה בחלונית ספירת מדריכים במודול השיבוץ).

## אילוצים
- יש מנגנון הרשאות קשיח לכמה תכונות לפי מזהה עובד (לא רק לפי Role).
- מודול השיבוץ תלוי ב-`ENABLE_SCHEDULING`; שינויי UI בלי פתיחת הפלאג לא ייראו למשתמשים.
- חלק מהלוגיקות משתמשות בנתוני Excel בצורת serial/date-number ולכן חשוב לא לשבור נרמול תאריכים.
- אם רוצים לבצע שינוי ממוקד (למשל “רק עמוד שיבוץ”), עדיף לעבוד רק על:
  - `scheduling.html`
  - `schedulingsystem.html`
  - `scheduling-app.js`
  - `core/*.js`
  בלי לגעת ב-`dashboard.js`/`app.js`.

## איך מריצים/בודקים מקומית
- המערכת היא Front-end סטטי עם fetch לקבצי JSON מקומיים, לכן מומלץ להריץ דרך שרת (לא לפתוח כ-`file://`).
- דרך מומלצת:
  - `node server.js`
  - גלישה ל-`http://localhost:5000`
- דפדפן מומלץ: Chrome/Edge עדכני.
- צעדי בדיקה בסיסיים:
  1. בדיקת התחברות משתמש מדריך ומשתמש אדמין.
  2. בדיקת הופעת/הסתרת כפתורים לפי הרשאות.
  3. בדיקת מסך שיבוץ רק עם מזהים מורשים.
  4. בדיקת טעינת `scheduling.json` והצגת תוצאות.
